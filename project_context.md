# OneStream – Project Context & Technical Architecture

This document provides an exhaustive technical breakdown of the OneStream platform. It serves as the definitive reference for understanding the technologies, architecture, and logic used to build this intelligence curation tool.

---

## 1. Tech Stack Overview

The project is built entirely on modern web primitives and utilizes an explicit set of frameworks, libraries, and SDKs optimized for a server-side rendered application:

*   **Core Framework**: Next.js 16.2.4 (App Router architecture)
*   **Frontend Library**: React 19.2.4 & React DOM 19.2.4
*   **Compilation/Build**: Leveraging Babel Plugin React Compiler (`babel-plugin-react-compiler`) and standard Next.js build tools.
*   **Styling Engine**: Tailwind CSS v4 alongside `@tailwindcss/postcss` for utility-class CSS and responsive design.
*   **State Management**: Zustand v5 (`zustand`), specifically utilizing the `persist` middleware for LocalStorage synchronization.
*   **Iconography**: Lucide React (`lucide-react`) for consistent, scalable SVG icons across the interface.
*   **Data Parsing**: `rss-parser` v3 for server-side processing of generic RSS and Atom XML feeds.
*   **AI/LLM Engine**: Groq SDK (`groq-sdk`), integrating the `llama-3.1-8b-instant` model for real-time natural language content categorization and article summarization.
*   **Typography**: Next.js native font optimization (`next/font/google`) using `Inter` for body text, `Manrope` for UI headlines, and `Space_Grotesk` for specialized futuristic branding.

---

## 2. Architecture & Directory Structure

The application strictly follows the **Next.js App Router** design pattern, centralizing routing in the `src/app` directory while maintaining separation of concerns for shared logic.

*   `src/app/` – Contains all page routes, nested layouts, and backend API handlers.
    *   `src/app/page.tsx` – The root landing page. A highly customized, animated canvas portal.
    *   `src/app/dashboard/` – The authenticated/main application shell. Contains its own `layout.tsx` which injects a persistent sidebar and mobile navigation over its child pages (`/feed`, `/sources`, `/saved`, `/settings`).
*   `src/app/api/` – Serverless routes acting as an orchestration layer to fetch external data and interact with AI endpoints, bypassing CORS limitations.
*   `src/components/` – Pure, reusable React UI components.
    *   `FeedCard.tsx`, `CompactFeedCard.tsx`, `SourceCard.tsx`, `AddSourceModal.tsx`, `ThemeProvider.tsx`.
*   `src/store/` – Zustand state management logic (`useSourceStore.ts`).
*   `src/lib/` – Constants, utility functions, and mock databases. Contains `sourceCatalog.ts` which acts as the definitive static list of available RSS URLs mapping to unique IDs.
*   `public/` – Static assets.

## 3. Core Components Breakdown

### The Homepage Portal (`src/app/page.tsx`)
A complex "Server Component" by default but utilizing a massive `use client` directive. It renders:
1.  **Canvas Animation (`useNetworkCanvas`)**: A highly optimized HTML5 `<canvas>` effect drawing 80 orbital data points, calculating velocity, alpha decays, and interactive hovering states based on mouse intensity.
2.  **Portal Tiles (`PortalTile`)**: Interactive `<Link>` blocks that mimic glassmorphism using raw Tailwind classes (`backdrop-blur-xl`, `bg-[#14151f]/70`). These house nested "Mini" components (e.g., `MiniGraph`, `MiniHoloCards`) that provide animated previews of the destination pages.

### Dashboard Shell (`src/app/dashboard/layout.tsx`)
The structural envelope holding the primary app UI.
*   Renders a desktop-fixed `aside` sidebar and a mobile-optimized sliding menu controlled by a local `useState` boolean (`sidebarOpen`).
*   Uses `usePathname()` from `next/navigation` to dynamically highlight the active route in the sidebar map.

### Feed Elements (`FeedCard.tsx` & `CompactFeedCard.tsx`)
The visual representation of an RSS item. 
*   **Props**: Takes normalized article shapes generated by the API (`title`, `excerpt`, `link`, `sourceName`, `sourceIconType`, `categories`, `timestamp`, `thumbnail`).
*   **Interactivity**: Contains local state for an "AI Summary" drawer. When toggled, it hits the `/api/summarize` endpoint, passing the article title and excerpt, turning the LLM's response into a dropdown brief.

### Source Modal (`AddSourceModal.tsx`)
A sliding modal over the screen that allows users to discover and subscribe to new RSS feeds. It queries `/api/sources/search` to pull data from the source catalog and uses Zustand's `addSourceById` function to inject it into local state.

## 4. State Management & Data Flow

Data persistency and UI reactivity rely heavily on **Zustand**.

1.  The `useSourceStore.ts` file instantiates a global store wrapped in `persist`. It is configured to save exactly four keys to the browser's `localStorage`: `sources`, `bookmarks`, `theme`, and `cacheDuration`.
2.  **Initializing**: On load, it builds a `defaultSources` list mapping standard IDs to objects defined in `SOURCE_CATALOG` to populate new user screens.
3.  **Data Flow to the Feed**:
    *   The user navigates to `/dashboard/feed`.
    *   A `useEffect` hook reads `sources` from Zustand and filters for items where `enabled === true`.
    *   It extracts the IDs of these active sources and appends them to a fetch request: `/api/feed?sources=hn,techcrunch...`
    *   To prevent excessive fetches when navigating back and forth, the store acts as a local frontend cache (storing `feedCache` and validating against `lastFetchTime` + `cacheDuration`).

## 5. Backend & API Logic

All backend operations happen within Next.js API Routes, effectively acting as lightweight serverless functions.

### `GET /api/feed`
The most complex route in the system, acting as an RSS aggregator and AI orchestrator.
1.  **Ingestion**: Receives a comma-separated list of `sources` IDs. Looks up their hardcoded XML URLs via `getCatalogFeedUrl()`.
2.  **Concurrency & Resilience**: Maps the URLs into an array of `rss-parser` promises. These promises are wrapped in a generic `fetchWithTimeout` constraint set to 7 seconds. If an RSS feed host is slow or down, it returns an empty array rather than failing the whole batch.
3.  **Normalization**: Custom logic `decodeEntities` and `stripHtml` cleans messy XMLCDATA tags into safe strings. `extractThumbnail` runs a cascade of fallbacks (from `media:content`, `media:thumbnail`, `enclosure`, to raw `<img>` regex slicing) to find an article image.
4.  **AI Batch Categorization**: For the top 20 aggregated articles, it maps out a minimal JSON `{ index: title }` structure to send to **Groq**. The model (`llama-3.1-8b-instant`) processes this as a single strict JSON system prompt (`response_format: { type: "json_object" }`). The endpoint extracts the tags and injects them back into the articles array before transmission to the frontend. Note: If the Groq API key is missing or the request fails, it gracefully falls back to assigning a default `["News"]` tag.

### `POST /api/summarize`
Handles on-demand article briefings.
*   Takes a `title` and `excerpt`.
*   Connects to Groq with a strict system prompt acting as a "senior intelligence analyst".
*   Yields a concise 3-4 sentence text block back to the `FeedCard`.

### `GET /api/sources/search`
A purely internal filtering utility. It takes a `?q=` user search string and a `?exclude=` string (IDs the user already has) and returns matching objects structurally defined in the local `sourceCatalog.ts` file.

## 6. Unresolved Architecture / Mocked Areas

While the app functions as a sophisticated feed aggregator, certain architectural portions are artificially mocked or run purely in local memory:

*   **Authentication & Database**: There is no actual backend database (like PostgreSQL or MongoDB). There is no user authentication. The "Create Account" buttons on the landing page are superficial links that redirect straight to the dashboard. All "user profiles" live entirely in the user's LocalStorage.
*   **Most Active Sources / Trending Topics Widgets**: Located in the feed sidebar, these are dynamically derived algorithms running on the *currently fetched* batch of 20-30 articles. They are not global statistics encompassing a historical database.
*   **Settings Implementation**: In the settings panel, options for Theme, Local Groq API Key, and Background Cache Duration are directly wired into Zustand and functional. However, other rendered texts—like "Notifications: On"—are purely visual placeholders with no backend wiring or Push API implementations.
*   **Saved Articles / Bookmarks**: Articles saved to "Saved Articles" rely precisely on the state representation of the article at the time of clicking "Save". If the article is deleted from the source website, the bookmark will persist locally with its cached thumbnail and excerpt.
